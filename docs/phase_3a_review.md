## Critical question — rho seq-sep mystery (RESOLVED)

### Forensic summary

The reference C++ source in `docs/reference_lammps_awsem/fix_backbone.cpp` does **not** contain a code path that yields `min_seq_sep=12` for the dumped `rho_i, rho_j`:

| C++ site | rho is | filter |
|---|---|---|
| `compute_decoy_ixns` line 5270-5271 | `well->ro(rand_resno)` | (via `well->ro`) |
| `compute_tert_frust` line 5092-5093 (written to dump line 5104) | `well->ro(i_resno)` | (via `well->ro`) |
| `cWell::ro(i)` (smart_matrix_lib.h:548-557) | memoized call to `compute_ro(i)` | — |
| **`cWell::compute_ro(i)`** (`smart_matrix_lib.h:630-652`) | sum over j with `chain_no[i]!=chain_no[j] || abs(res_no[j]-res_no[i])>1` | **`min_seq_sep=1`** |

So in the upstream LAMMPS-AWSEM source, both burial energy AND the dumped rho should use `min_seq_sep=1`. Yet our `compute_rho(min_seq_sep=1)` matches V_burial at machine precision while the dump matches `min_seq_sep=12`.

### Resolution: frustratometeR ships a patched binary

The validation dump was generated by the **frustratometeR** R package (`tertiary_frustration.dat` + the post-processed `*_configurational.dat` with `Welltype` / `FrstState` columns is the frustratometeR output format, not raw LAMMPS-AWSEM).

frustratometeR's `R/functions.R` lines 276-285:
```
@param SeqDist Sequence at which contacts are considered to interact (3 or 12).
               Type: numeric. Default: 12.
@param SeqDist Sequence at which contacts are considered to interact (3 or 12).
```

and lines 427-432:
```r
system(paste("cp ", Pdb$scriptsDir, "/lmp_serial_", SeqDist, "_Linux ", ...
```

frustratometeR **ships two PRECOMPILED LAMMPS binaries** — `lmp_serial_3_*` and `lmp_serial_12_*` — where the integer 3 or 12 is **compiled into the binary as the minimum sequence separation for the rho computation**. The R wrapper picks whichever the user requested (default 12). The on-disk `fix_backbone.cpp` reference in our repo is the upstream LAMMPS-AWSEM source — it does NOT match the frustratometeR-bundled binary.

(GitHub: `proteinphysiologylab/frustratometeR/blob/master/R/functions.R` lines 276, 285, 302-303, 427-428. Confirmed via `gh api`.)

### Verification

Direct empirical test on 5AON:

| min_seq_sep | rho_1 | rho_2 | rho_3 | rho_6 | rho_7 |
|---|---|---|---|---|---|
| 1 (burial) | 1.266 | 1.461 | 3.092 | 6.071 | 6.254 |
| 12 (frustratometeR dump) | 0.000 | 0.000 | 0.304 | 2.791 | 2.999 |
| LAMMPS dump (5AON tertiary_frustration.dat) | 0.000 | 0.000 | 0.304 | 2.791 | 2.999 |

The dump matches `min_seq_sep=12` **exactly** (max diff 4.7e-4 across all 49 residues; floating-point precision of `tanh`/`sqrt`).

### Is the workaround correct?

**Yes.** The agent's `lammps_dump_rho()` helper with `LAMMPS_DUMP_RHO_MIN_SEQ_SEP=12` is correct **for matching frustratometeR's default SeqDist=12 binary**. Two important caveats the docstring should add (see Medium-1 below):

1. The "12" is **not a bug in upstream LAMMPS-AWSEM** — it's a configuration choice baked into frustratometeR's bundled `lmp_serial_12_*` binary. If a user ever validates against `lmp_serial_3_*` (SeqDist=3) or against an OpenAWSEM Python run (which uses `min_sequence_separation_rho=2` by default), the value must change.
2. The choice of 12 is principled. From the AWSEM-MD literature, `min_seq_sep=12` is the standard "long-range contact" threshold (the same value used to count "topological contacts" — non-local in sequence but local in space). frustratometeR adopted it deliberately so the decoy ensemble only samples truly tertiary (non-local) contacts. The energy/burial calculation keeps `min_seq_sep=1` because that's the underlying physical rho. **The dump is consistent with itself**: water energy uses `(i,j)` pairs with `|i-j|>=12` (frustratometeR `contact_cutoff` patch), and the `rho` displayed alongside is the rho the DECOY formula actually sampled from a per-residue pool that EXCLUDES local-in-sequence contributions.

### Recommendation

Promote `LAMMPS_DUMP_RHO_MIN_SEQ_SEP=12` to a function parameter on `configurational_decoy_stats` and `lammps_dump_rho`, default 12, so callers validating against `lmp_serial_3_*` or OpenAWSEM can pass `min_seq_sep_rho=3` or `=2` without code edits. (Medium-2 below.)

### Medium

**M-1 — `decoys.py` docstring blames a "subtle bug" in upstream LAMMPS.** Lines 8-38 of `src/decoys.py` describe the rho discrepancy as "the LAMMPS source has a hidden subtle bug or undocumented filter that effectively applies min_seq_sep=12". This is **incorrect** — the upstream source applies `min_seq_sep=1` and there is no bug. The frustratometeR binary is a deliberate patch (`lmp_serial_12_Linux`) chosen by the wrapper, controlled by the R parameter `SeqDist` (default 12, only 3 or 12 valid). The docstring should be reworded:
- "The dump was generated by frustratometeR's bundled `lmp_serial_12_Linux` binary, which applies `SeqDist=12` to the rho computation (deliberate, not a bug)."
- Mention that `SeqDist=3` and `OpenAWSEM` (`min_sequence_separation_rho=2`) would NOT match.
- Cite `frustratometeR/R/functions.R:276,285,302,427-432` for the evidence.

**M-2 — `min_seq_sep_rho` should be a function argument on the user-facing helpers.** Currently the value 12 is hard-coded in `LAMMPS_DUMP_RHO_MIN_SEQ_SEP` and the helper `lammps_dump_rho` exposes `min_seq_sep` (good) but `configurational_decoy_stats` does NOT — it always calls `lammps_dump_rho(coords, device=device)` with the default. To make the module work with SeqDist=3 binaries or with frustrapy/OpenAWSEM dumps, expose `min_seq_sep_rho: int = 12` on `configurational_decoy_stats` and pass it through. Cheap; one extra param.

**M-3 — `_burial_total` has a tight Python loop over 3 wells when a stacked tensor op is one line.** Lines 621-635 in `compute_configurational_decoy_energy` build the per-well switches inside a `for w_idx in range(3)` Python loop and stack. This is fine at 1000 decoys (microseconds), but in Phase 3b mutational mode where this gets called N*N times per protein, the Python loop overhead becomes N²·3 = ~600k iterations on a 450-res protein. Recommend vectorising:
```python
rho_min = torch.as_tensor(burial_rho_min, dtype=dtype, device=device)   # (3,)
rho_max = torch.as_tensor(burial_rho_max, dtype=dtype, device=device)   # (3,)
t_min = torch.tanh(kappa_t * (rho_vec.unsqueeze(-1) - rho_min))        # (n_decoys, 3)
t_max = torch.tanh(kappa_t * (rho_max - rho_vec.unsqueeze(-1)))        # (n_decoys, 3)
return -0.5 * k_burial_t * (bg_per_decoy * (t_min + t_max)).sum(-1)
```
Same result, no Python loop. (Not required for Phase 3a correctness; flag for Phase 3b performance.)

### Low / nit

**L-1 — `compute_configurational_decoy_energy` accepts `gamma_mediated_protein=None, gamma_mediated_water=None` independently** (lines 562-567), which is helpful for ablations. But the implicit cached-load fallback gets BOTH from `_cached_load_mediated_gamma`. If a user passes `gamma_mediated_protein=X, gamma_mediated_water=None`, the cached water gets loaded — fine, but worth a comment that the loader returns a TUPLE so partial overrides do work.

**L-2 — Module-level cache key uses `str(device)` which is fragile across CUDA device indices.** `torch.device("cuda")` and `torch.device("cuda:0")` are functionally the same on a single-GPU box but stringify differently. Not a bug in practice (`_dtype_to_str` is called consistently), but if Phase 4+ ever uses multi-GPU, normalise via `torch.device(device).type + (":" + str(torch.device(device).index) if torch.device(device).index else "")` or just `device.type`.

**L-3 — `sample_configurational_decoys` rejection-sampler fallback (lines 426-449) tiles existing in-contact distances** to fill the deficit. This biases the rij distribution toward whichever pairs happen to satisfy the cutoff first. The comment says "this branch is informational only — never hit on the panel proteins" which is true for typical folded proteins, but for very sparse / fragmented inputs (single helix, dimers separated by gaps) it could silently activate. Recommend logging a warning when `filled < n_decoys` instead of silently tiling.

## Things I verified are correct

1. **Configurational caching pattern (`already_computed_configurational_decoys` gate, fix_backbone.cpp:5341)** — `configurational_decoy_stats` is structured as a one-shot computation per call; the caller is responsible for caching the result across pairs. The test `test_configurational_cache_one_stat_per_protein` confirms identical means/stds for repeated same-seed calls. The "cache" is at the architectural level (don't call twice in the inner loop); the function itself is stateless. Correct.

2. **AA sampling: uniform residue-index → AA composition follows protein.** Lines 389-392 use `torch.randint(0, n, ...)` then index into `coords["residue_types"]`, exactly matching C++ `get_random_residue_index() → get_residue_type(rand_resno)` (fix_backbone.cpp:5281-5284, 5550-5554, 5585-5588). NOT 1/20 uniform — correct.

3. **rij rejection sampling: reject `rij >= cutoff OR p == q`.** Line 419 implements `(p_dev != q_dev) & (cand_dist < contact_cutoff)`, matching `while(rij > tert_frust_cutoff || rand_i_resno == rand_j_resno)` (fix_backbone.cpp:5262). The `||` in C++ becomes `&` on the ACCEPT mask after De Morgan — correct.

4. **rho sampling: independent random pair draws (no contact constraint).** Lines 395-398 draw a fresh pair of indices for rho_i and rho_j, matching fix_backbone.cpp:5268-5271 ("get new pair of random residues for burial term"). Correct.

5. **Energy formula:** E_decoy = V_water (direct + mediated) + V_burial(aa_i, rho_i) + V_burial(aa_j, rho_j), no DH. Matches fix_backbone.cpp:5287-5295 (electrostatic_energy=0 when huckel_flag is off; frustratometeR's `awsem.in` does set huckel_flag, but only if the user requested electrostatics — the default panel runs do NOT). The decoy formula correctly OMITS electrostatics by not even providing a knob for it. If Phase 3b ever adds DH-aware decoys, this needs revisiting.

6. **Caching helpers:** `_cached_load_mediated_gamma`, `_cached_load_direct_gamma`, `_cached_load_burial_gamma` are all `@functools.lru_cache(maxsize=8)` (lines 169-187). Phase 2b latent fix #2 is correctly applied. Test `test_gamma_cache_at_module_level` verifies cache hit on repeat call.

7. **Gamma indexing contract:** Lines 579-581 use `gamma_direct[aa_i, aa_j]` with both `aa_i` and `aa_j` as 1-D tensors of length `n_decoys`. PyTorch advanced indexing semantics: this is **elementwise gather** (length `n_decoys`), NOT outer product. The docstring "Contract on inputs" (lines 102-111) is explicit and the regression test `test_decoy_gamma_indexing_is_elementwise_not_outer` proves it. Phase 2b latent fix #3 is applied.

8. **n=0 / n=1 early-out includes gamma_pair key.** Verified in `src/water_mediated.py:217-225` — the empty-result dict includes `"gamma_pair": empty_mat`. Test `test_water_mediated_early_out_has_gamma_pair` verifies for both n=0 and n=1. Phase 2b latent fix #1 is applied. Note: the decoy module itself raises ValueError on `n<2` (correct — decoys are undefined for monomers), which is tested in `test_decoy_raises_on_n_lt_2`.

9. **Random seed reproducibility.** `torch.Generator(device="cpu")` is used for all four random draws (aa_i, aa_j, rho_i, rho_j, then the rejection-sampler pair `(p, q)`). Same seed → identical histograms. Test `test_configurational_cache_one_stat_per_protein` shows five same-seed runs produce identical results to 1e-9. Note: CPU generator is used even on GPU paths (intentional — see docstring lines 336-339, which calls this out for cross-device test reproducibility).

10. **GPU/CPU agreement.** Verified locally on RTX 4070: 5AON `configurational_decoy_stats(seed=0, float64)` gives identical (mean, std) on both within 4e-9 (population-std numerical noise). No symptoms of CUDA-specific drift.

11. **Population std (ddof=0).** Line 644: `decoy_energies.std(unbiased=False)` — matches LAMMPS `compute_array_std` (divides by N, not N-1). Verified by reading the C++ around the std computation; ddof=1 (sample std) would give a different number for n=1000 (factor √(1000/999) ≈ 1.0005, just inside our 3% tolerance but the wrong direction).

12. **NaN-safe distance handling** (lines 367-382): rows with NaN coordinates are replaced with `1e6` before distance computation, then any NaN-poisoned pair is forced to `+inf`. Both ensure the rejection criterion `< cutoff` is False for missing residues. Defensive and correct.

## Recommended Phase 3b next step

Phase 3b (mutational mode) will add a per-pair decoy generator that:
- Holds `rij, rho_i, rho_j` at the NATIVE values (from `compute_tert_frust`'s line 5092-5093 — these are `well->ro(i_resno)`, i.e. ours/`burial.py`'s `min_seq_sep=1` rho, NOT the dumped rho).
- Scrambles ONLY `aa_i, aa_j`.
- Adds `(i, k)` and `(j, k)` cross-terms for all `k != i, j` with `rik < cutoff` or `rjk < cutoff` (C++ lines 5300-5327).
- Uses `rho_k = get_residue_density(k) = well->ro(k)` for the cross-term burial.

### Latent issues that will only surface in Phase 3b

1. **Native rho vs dump rho — pick one and document.** In mutational mode, the C++ uses the SAME `well->ro(i_resno)` for both the (i,j) anchor and the (k) cross-terms (fix_backbone.cpp:5306, 5092). frustratometeR's `lmp_serial_12_*` patches this to use `min_seq_sep=12`. So when validating against frustratometeR `mutational` dumps:
   - Native rho_i, rho_j held constant per pair → **must use the SAME min_seq_sep_rho=12 as in Phase 3a**, NOT the burial `min_seq_sep=1`.
   - Cross-term rho_k → ALSO uses `min_seq_sep=12` (same `well->ro` getter).
   - V_burial(aa_k, rho_k) cross-terms therefore use `min_seq_sep=12` rho. This is a deviation from the conventional V_burial which uses `min_seq_sep=1`. **The mutational E_decoy will NOT decompose into our standard V_burial + V_water at machine precision** — they live on different rho domains.

   **Suggested check:** before writing the mutational driver, compute V_burial on `lammps_dump_rho(coords)` (min_seq_sep=12) for 5AON and confirm it equals neither -41.799 (the energy.log value with min_seq_sep=1) nor what frustratometeR reports for the per-residue burial term. This will tell you whether to call this a "frustratometeR-specific dump-rho V_burial" branch or expose another flag.

2. **Cross-term contact mask must match `compute_tert_frust`'s inner loop, not the outer one.** The outer `compute_tert_frust` loop (line 5086) uses `(abs(i-j)>=contact_cutoff || i_chno != j_chno)` to decide whether to ANALYZE the pair. The inner cross-term loop (line 5300-5327) uses ONLY `rik < tert_frust_cutoff` (spatial), NO sequence-separation filter. This is asymmetric. A naive Phase 3b might apply the seq-sep filter to cross-terms too — that would be wrong. Make the test gate this explicitly with a paper-thin protein (e.g. a 30-residue helix) where every (i, k) pair has small `|i-k|`.

3. **Mutational re-runs gamma loaders N²/2 times if cache miss.** With the lru_cache on `_cached_load_mediated_gamma`, this is fine after the first call. But if Phase 3b inadvertently passes `gamma_mediated_protein=X, gamma_mediated_water=Y` directly (bypassing the cache), the `.to(device, dtype)` cast on lines 568-573 still copies the tensor N²/2 times. Solution: do the cast ONCE outside the inner loop and feed pre-cast gammas. Cheap structural fix.

4. **Per-pair decoy generation is the expensive path.** Phase 3a was 1000 decoys × 1 call. Phase 3b will be 1000 decoys × O(N²/2) pairs × the cross-term expansion (~20-30 contacts per pair). For 11BG (248 res, ~5,500 pairs in the dump) that's ~5.5M decoy evaluations. Vectorise across decoys (already done) AND across pairs if possible. The `_burial_total` Python loop flagged in M-3 above becomes the hot path here. Strongly recommend fixing M-3 before Phase 3b.

5. **`compute_water_energy` is called THREE times per decoy in mutational mode** (once for (i,j), once for each (i,k), once for each (j,k)). Each call evaluates the same sigmoid windowing with different gamma table lookups. If you reuse the `theta_direct, theta_mediated` pre-computation from native, you can halve the call count. Optional optimisation.

6. **DH still gated off in mutational mode** (assuming frustratometeR default). Same reasoning as Phase 3a. If the user ever passes `huckel_flag=true`, mutational E_decoy gains an `electrostatic_energy += compute_electrostatic_energy(rik, ...)` + `compute_electrostatic_energy(rjk, ...)` term (fix_backbone.cpp:5315-5325). Worth gating this behind a `include_dh: bool = False` parameter on the mutational driver, even if always False today.

### Phase 3b suggested first step

Before writing the driver, **build a single ground-truth mutational test case**:
1. Pick one (i, j) pair from `5AON_tertiary_frustration.dat` where mutational mode (lmp_serial_12 with `Mode=mutational`) exists. (Re-run frustratometeR with `Mode="mutational"` if no mutational dump is in the benchmark dir.)
2. Re-implement the C++ mutational decoy formula in plain Python for THAT one pair, verifying it reproduces the dumped (mean, std) for that pair.
3. Then write the vectorised PyTorch version and assert it matches that ground truth.

This sidesteps the per-pair caching question entirely and pins the rho/cross-term semantics with a numerical witness before any optimisation.
